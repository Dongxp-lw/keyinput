# 信息架构设计

本文档定义 Private Input Vault 的产品信息架构。它承接 [产品决策记录：主动选择式私密输入](product-decision-record.md) 和 [产品整体设计](product-design.md)，用于说明产品中的对象、入口、导航层级、状态关系和版本分层。

本文档不定义具体视觉布局、线框图、组件样式或平台实现代码。后续交互设计可以在本文档基础上继续细化关键路径。

## 1. 信息架构目标

信息架构必须帮助用户形成一个稳定心智：这是一个本地加密保险库，用户从中主动选择要输入的私密字段；平台凭据和自动填充能力是标准登录场景的兼容增强，不是产品主路径。

信息架构需要达成以下目标：

- 让用户清楚区分“管理保险库”和“输入私密内容”两个任务空间。
- 让用户在主应用、安全键盘和平台凭据入口之间保持同一个条目/字段心智。
- 让离线本地能力始终位于一级结构中，而不是藏在账号或云服务之后。
- 让备份、迁移、恢复和导出这些数据控制能力容易找到。
- 让平台限制、自动填充状态和权限引导有明确位置，但不抢占主动选择主路径。
- 让后续云同步、订阅、家庭/团队能力可以增量加入，而不重写核心结构。

## 2. 调研转译

外部调研用于校准结构，而不是替产品决定方向。

### 2.1 用户心智与信息架构

Nielsen Norman Group 对 mental models 的总结强调：用户会基于已有经验预测系统如何工作，信息架构需要尽量贴合这种心智；如果产品有新概念，就必须用清晰标签和结构帮助用户建立正确模型。

对本产品的转译：

- 用户熟悉“保险库”“条目”“字段”“搜索”“设置”“导出”这些概念，应优先使用这些词。
- “主动选择”是本产品的新心智，必须在入口和流程中反复体现。
- 不应让用户误以为产品会自动判断目标应用或网站是否可信。
- 不应让平台自动填充入口看起来比本地保险库和安全键盘更核心。

### 2.2 平台凭据入口

Android Credential Manager Provider 使用查询和选择两个阶段：系统先请求可用凭据，用户选择后再进入 provider activity 完成认证和返回结果。Android 还提供凭据提供方设置入口。iOS Password AutoFill 支持第三方密码管理器通过 AutoFill Credential Provider Extension 提供凭据和 TOTP。

对本产品的转译：

- 平台凭据能力在信息架构中应是“平台兼容与填充设置”，而不是新的保险库中心。
- 平台入口必须能引导用户解锁，但不能在锁定状态下泄露明文。
- 平台入口要有状态页：是否启用、支持哪些类型、哪些场景受限、如何回到安全键盘或主应用。
- 条目必须能保存平台匹配信息，但这些信息不能被解释为目标身份真实性判断。

### 2.3 移动导航原则

移动端结构应减少同层入口数量，避免多个相似入口竞争同一个任务。对本产品来说，主应用不应同时出现多个“填充”“输入”“键盘”“自动填充”入口而让用户困惑。

对本产品的转译：

- 一级导航最多承载少量长期任务区域。
- 平台兼容能力归入设置/状态入口，不作为与保险库并列的主要工作空间。
- 安全键盘是独立输入面，不把主应用的全部管理能力搬进键盘。
- 搜索入口要有明确作用域：主应用搜索保险库，安全键盘搜索可输入内容。

## 3. 用户心智模型

产品应让用户形成四个核心心智对象。

| 心智对象 | 用户理解 | 产品承诺 |
| --- | --- | --- |
| 保险库 | 我的私密数据存放处。 | 本地加密，不登录也能使用。 |
| 条目 | 一个账号、身份、备注或私密资料组。 | 可搜索、可编辑、可导出、可迁移。 |
| 字段 | 条目里某个具体可输入的值。 | 用户明确选择后才插入或复制。 |
| 输入面 | 我把字段送到其他应用的地方。 | 安全键盘为主，平台填充为兼容增强。 |

平台凭据、云同步、订阅、设备管理都应被理解为附加能力，而不是用户理解产品的起点。

## 4. 顶层信息架构

主应用一级结构建议如下：

```text
Private Input Vault
  -> 保险库
  -> 生成器
  -> 备份与迁移
  -> 安全设置
  -> 平台兼容
  -> 云与设备（后续启用）
```

一级结构说明：

| 一级区域 | 是否首屏核心 | 离线可用 | 说明 |
| --- | --- | --- | --- |
| 保险库 | 是 | 是 | 管理条目、字段、搜索、标签、收藏。 |
| 生成器 | 是 | 是 | 生成密码、后续生成私密字段模板。 |
| 备份与迁移 | 是 | 是 | 导出、导入、本地迁移、恢复。 |
| 安全设置 | 是 | 是 | 主密码、生物识别、自动锁定、剪贴板、屏幕保护。 |
| 平台兼容 | 否 | 部分 | 自动填充、Credential Provider、键盘启用状态和限制说明。 |
| 云与设备 | 否 | 否 | 登录、同步、设备管理、版本历史、订阅。 |

“平台兼容”可以在设置中呈现，也可以在首轮引导中出现，但不应替代“保险库”和“安全键盘”的核心位置。

## 5. 对象模型

信息架构围绕以下对象组织。

```text
Vault
  -> Entry
      -> Field
      -> Tag
      -> PlatformMatch
      -> UsagePolicy
  -> BackupPackage
  -> DeviceRecord
  -> Settings
```

### 5.1 Vault

Vault 是本地保险库。用户不需要理解文件格式，但需要理解：保险库归自己所有，主密码是根秘密，忘记主密码不能由服务端恢复。

Vault 的主要入口：

- 首次创建。
- 解锁。
- 自动锁定后重新进入。
- 导出、导入和迁移。
- 安全设置。

### 5.2 Entry

Entry 是用户管理的主单位。一个 Entry 可以是登录账号、安全备注、身份资料、支付资料、恢复码集合、地址模板或自定义私密文本组。

Entry 的主要入口：

- 保险库列表。
- 搜索结果。
- 标签和收藏。
- 安全键盘搜索结果。
- 平台凭据匹配结果。
- 导入预览和冲突处理。

### 5.3 Field

Field 是实际输入单位。主动选择模型必须落在 Field 层，而不是只落在 Entry 层。

字段需要表达：

- 字段类型：用户名、密码、TOTP、邮箱、电话、地址、文本、备注、恢复码、自定义字段。
- 敏感级别：普通、敏感、高敏感。
- 输入行为：插入、复制、仅显示。
- 认证策略：是否需要重新认证。
- 平台可见性：是否允许出现在系统填充建议中。

### 5.4 PlatformMatch

PlatformMatch 描述条目与应用、网站或平台凭据请求之间的关系。它只能用于匹配和提示，不能作为“目标可信”的证明。

可包含：

- 域名或关联域。
- Android 包名和签名相关信息。
- iOS App ID 或关联域信息。
- 浏览器来源信息。
- 用户手动确认的别名或备注。
- 匹配置信等级和来源。

### 5.5 UsagePolicy

UsagePolicy 描述字段如何被使用。

可包含：

- 是否允许安全键盘插入。
- 是否允许剪贴板复制。
- 是否允许平台凭据填充。
- 是否要求重新认证。
- 是否显示高敏感提醒。
- 是否隐藏在搜索预览中。

## 6. 主应用结构

### 6.1 保险库

保险库是主应用默认工作区。

```text
保险库
  -> 搜索
  -> 全部条目
  -> 收藏
  -> 标签
  -> 最近使用
  -> 已归档
  -> 条目详情
```

设计规则：

- 搜索应默认覆盖条目标题、字段标签、标签和类型，不默认搜索字段明文，除非用户已解锁且安全策略允许。
- 收藏用于高频主动输入，不暗示目标应用或网站可信。
- 最近使用应可关闭或清除，避免泄露使用习惯。
- 已锁定时不展示条目标题和最近使用，除非后续安全评审允许低敏摘要。

### 6.2 条目详情

```text
条目详情
  -> 基本信息
  -> 字段列表
  -> 标签与收藏
  -> 平台匹配
  -> 使用策略
  -> 历史/恢复（后续）
```

设计规则：

- 字段列表必须清楚区分“显示值”“复制值”“插入值”。
- 高敏感字段默认隐藏，显示、复制、插入都可触发重新认证。
- 平台匹配信息应放在条目详情的次级区域，不应让用户误解为安全认证。
- 使用策略应让用户控制字段是否出现在安全键盘和平台填充中。

### 6.3 生成器

```text
生成器
  -> 密码生成
  -> 密码短语生成
  -> 用户名/别名生成（后续）
  -> 私密字段模板（后续）
```

设计规则：

- 生成器离线可用。
- 生成结果默认不写入日志、不上传、不自动保存。
- 保存到条目必须由用户明确选择。

### 6.4 备份与迁移

```text
备份与迁移
  -> 加密导出
  -> 加密导入
  -> 本地设备迁移
  -> 导入预览
  -> 冲突处理
  -> 恢复说明
```

设计规则：

- 备份和导出是核心数据控制能力，不应被云同步取代。
- 导入前必须展示结果类型：新增、合并、替换、冲突。
- 导入失败不得破坏现有保险库。
- 用户必须知道导出包仍需要主密码或恢复材料才能解密。

### 6.5 安全设置

```text
安全设置
  -> 主密码
  -> 生物识别/设备凭据
  -> 自动锁定
  -> 剪贴板
  -> 屏幕与预览保护
  -> 日志与诊断
  -> 高敏感字段策略
```

设计规则：

- 主密码恢复边界必须清晰可见。
- 生物识别是便利机制，不应被描述为主密码替代品。
- 剪贴板能力必须有自动清除和风险说明。
- 诊断默认不应上传秘密、字段值、TOTP 种子或生成码。

### 6.6 平台兼容

```text
平台兼容
  -> 安全键盘启用状态
  -> 自动填充/凭据提供方状态
  -> TOTP 填充状态
  -> 支持与限制
  -> 系统设置引导
  -> 兜底路径说明
```

设计规则：

- 平台兼容是状态和引导区域，不是主工作区。
- 每个平台只展示当前可用能力，不承诺跨平台完全一致。
- 系统设置跳转应说明跳转后由系统控制。
- 如果平台填充失败，应明确指向安全键盘、复制或主应用路径。

### 6.7 云与设备

```text
云与设备
  -> 登录
  -> 同步状态
  -> 设备列表
  -> 版本历史
  -> 订阅权益
  -> 家庭/团队（后续）
```

设计规则：

- 未登录时，本地保险库不应降级。
- 退出登录不删除本地保险库，除非用户明确选择。
- 云同步状态不得阻断本地编辑、输入、导出和导入。
- 云端只存储密文这一点必须在相关入口中保持一致表达。

## 7. 安全键盘结构

安全键盘是主动输入面。它不是主应用的缩小版。

```text
安全键盘
  -> 锁定态
  -> 解锁态
      -> 搜索
      -> 收藏
      -> 最近
      -> 条目详情
      -> 字段选择
      -> 兜底操作
```

### 7.1 锁定态

锁定态只允许用户识别产品、解锁或切换键盘。

锁定态不展示：

- 条目标题。
- 字段标签。
- 最近使用。
- 搜索历史。
- 任何秘密或敏感元数据。

### 7.2 解锁态

解锁态是短时授权状态。它应优先展示可输入内容，而不是管理功能。

可展示：

- 搜索入口。
- 收藏条目。
- 最近条目。
- 字段列表。
- 插入、复制、打开主应用。

不应承载：

- 创建复杂条目。
- 大量字段编辑。
- 云账号管理。
- 订阅购买。
- 大段安全说明。

### 7.3 字段选择

字段选择是安全键盘最重要的信息层级。

字段需要按输入价值排序：

1. 用户名/账号。
2. 密码。
3. TOTP。
4. 常用自定义字段。
5. 地址、备注、恢复码等其他字段。

高敏感字段可以要求二次确认或重新认证。字段被插入后，键盘应尽快清理短时敏感值。

### 7.4 兜底操作

兜底操作包括：

- 复制字段。
- 打开主应用。
- 切换到系统键盘。
- 查看平台限制说明。

兜底操作只在需要时出现，不应占据安全键盘的主要空间。

## 8. 平台凭据兼容结构

平台凭据兼容结构不是用户日常管理入口，而是系统调用和设置状态入口。

```text
平台凭据兼容
  -> 系统请求
  -> 锁定判断
  -> 用户认证
  -> 凭据候选
  -> 用户选择
  -> 返回结果
```

信息架构规则：

- 候选项来自保险库条目，但展示应遵循平台 UI 约束。
- 用户选择前，不应返回明文字段。
- 如果系统请求来自标准登录字段，可以展示匹配候选。
- 如果匹配不确定，应降低置信表达，不暗示目标可信。
- 如果平台能力不可用，应转向安全键盘或复制路径。

平台兼容能力在主应用中的状态页应包含：

| 状态对象 | 内容 |
| --- | --- |
| 安全键盘 | 是否启用、如何启用、平台限制。 |
| 自动填充 | 是否启用、支持哪些字段、如何回退。 |
| 凭据提供方 | 是否启用、支持密码/Passkey/TOTP 的状态。 |
| 匹配规则 | 域名、应用包名、关联域和用户确认来源。 |
| 兜底路径 | 打开安全键盘、复制、打开主应用。 |

## 9. 搜索与筛选

搜索是跨入口能力，但不同入口的搜索作用域不同。

| 入口 | 搜索范围 | 约束 |
| --- | --- | --- |
| 主应用保险库 | 条目、字段标签、标签、类型。 | 已锁定不搜索明文秘密。 |
| 安全键盘 | 可输入条目和字段。 | 优先快速选择，不展示复杂过滤器。 |
| 导入预览 | 导入包中的条目摘要。 | 导入前不覆盖本地数据。 |
| 平台兼容 | 匹配规则和状态项。 | 不作为日常输入搜索。 |

搜索结果排序建议：

1. 收藏。
2. 精确标题匹配。
3. 标签匹配。
4. 最近主动使用。
5. 字段标签匹配。
6. 平台匹配候选。

“最近使用”必须可清除或关闭。

## 10. 状态与入口关系

| 状态 | 主应用 | 安全键盘 | 平台兼容 | 云与设备 |
| --- | --- | --- | --- | --- |
| 未创建保险库 | 创建保险库 | 引导打开主应用 | 不启用 | 不显示为前置任务 |
| 已锁定 | 解锁入口 | 解锁入口 | 只显示认证动作 | 可显示登录状态，不解锁保险库 |
| 已解锁 | 完整管理 | 短时输入 | 可返回候选 | 可同步密文 |
| 离线 | 完整本地能力 | 可输入 | 依赖平台本地能力 | 待同步 |
| 退出登录 | 本地能力不变 | 本地能力不变 | 本地候选不应被删除 | 云能力关闭 |
| 平台受限 | 展示限制与兜底 | 提供复制/打开主应用 | 展示不可用原因 | 不应要求登录解决本地问题 |

## 11. 版本分层

### v0.1 信息架构范围

- 保险库最小结构。
- 安全键盘锁定态、解锁态、搜索、字段选择。
- 平台兼容调研入口，不要求完整实现。
- 平台限制记录。

### v0.2 信息架构范围

- 完整本地保险库区域。
- 条目详情和字段策略。
- 生成器。
- 备份与迁移。
- 安全设置。
- 平台兼容状态页。

### v0.3 信息架构范围

- 本地设备迁移。
- 导入预览。
- 冲突处理。
- 迁移恢复说明。

### v1.0 信息架构范围

- 首平台发布质量结构。
- 平台限制和兜底路径完整闭环。
- 隐私、安全和备份恢复入口清晰。

### v1.1 信息架构范围

- 云与设备。
- 同步状态。
- 版本历史。
- 订阅权益。
- 登录后服务能力。

## 12. 不应出现的结构

为避免偏离产品基线，信息架构中不应出现以下结构：

- 把“登录账号”作为创建本地保险库的前置入口。
- 把“自动填充”放在比“保险库”和“安全键盘”更核心的位置。
- 把“平台匹配”描述为“目标安全认证”。
- 把“云同步”作为备份和迁移的唯一入口。
- 把普通键盘词典、在线词库或云输入法能力作为默认模块。
- 在安全键盘中塞入完整条目管理、订阅或团队协作功能。

## 13. 验收清单

后续交互设计和平台设计应使用以下清单验证是否符合信息架构：

- 用户是否能在不登录、不联网时找到并使用本地保险库？
- 用户是否能清楚知道自己正在选择哪个条目和字段？
- 安全键盘是否保持短时输入面，而不是变成复杂管理界面？
- 平台兼容入口是否被描述为增强能力，而不是主路径？
- 备份、导出、导入和迁移是否不依赖第一方云？
- 锁定状态是否避免泄露条目、字段和最近使用信息？
- 平台限制是否有清楚说明和可用兜底路径？
- 云与设备是否不会阻断本地保险库能力？

## 14. 下一步

完成信息架构后，下一份文档建议是“核心交互设计”。它应细化五条路径：创建保险库、保存条目、安全键盘输入、平台兼容填充、导出导入。
