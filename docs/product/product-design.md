# 产品整体设计

本文档定义 Private Input Vault 的整体产品设计。它位于产品需求和具体交互/技术方案之间，用来说明产品由哪些体验面组成、用户如何完成核心任务、不同模块如何协作，以及哪些设计边界必须长期保持稳定。

本文档必须服从 [产品决策记录：主动选择式私密输入](product-decision-record.md)。后续任何平台能力、外部调研或“更好方案”，都应先在这条产品基线下评估。

本文档不定义具体视觉稿、页面线框、平台实现细节或密码学算法参数。这些内容应在整体设计稳定后，再分别进入信息架构、交互设计、平台方案和安全实现设计。

## 1. 设计目标

Private Input Vault 要成为一个离线优先、用户主动选择的凭据与私密输入系统。它的核心不是替用户猜测应该填入什么，而是让用户明确选择要输入的条目和字段。平台凭据接口可以作为标准登录场景的兼容增强，但不能取代主动选择式私密输入。

整体设计必须达成以下目标：

- 用户可以在不登录、不联网的情况下创建、解锁、管理和使用本地保险库。
- 产品可以兼容 Android Credential Manager/Autofill、iOS AutoFill Credential Provider 等平台凭据入口，但这些能力必须服务主动选择和离线优先，而不是改变产品主路径。
- 用户可以通过安全键盘主动选择字段，并将选中的值插入当前输入框。
- 主应用负责长期管理，安全键盘负责短时输入，两者边界清晰。
- 产品清楚说明自身不判断目标应用或网站是否可信，也不承诺钓鱼检测。
- 备份、迁移和云同步都不能削弱本地离线使用能力。
- 云能力只能作为可选服务存在，服务端不能接触保险库明文。

## 2. 调研结论

外部调研的作用不是替产品决定主路径，而是帮助我们识别哪些平台能力可以增强体验。主流密码管理器通常会接入系统凭据能力、自动填充能力和自身管理界面；这些做法值得参考，但必须放在“离线优先、用户主动选择、不承诺判断目标可信”的产品背景下取舍。

调研依据包括：

- Android 官方将 Credential Manager 定位为推荐的凭据交换 API，并支持密码、Passkey、联合登录和第三方凭据提供方。
- Android Autofill Framework 明确支持密码管理器作为 Autofill Service，并且 Android 11 起支持 IME 内联显示自动填充建议；系统会保护建议内容，在用户选择前不向 IME 暴露私密数据。
- iOS Password AutoFill 支持第三方密码管理器通过 AutoFill Credential Provider Extension 提供用户名、密码和 TOTP。
- iOS 自定义键盘在安全输入框、电话键盘字段，以及拒绝第三方键盘的应用中会被系统键盘替换；启用 open access 又会显著增加用户信任负担。
- OWASP MASVS 将移动安全拆分为存储、密码学、认证、网络、平台交互、代码质量、抗逆向和隐私等控制组，适合作为后续安全设计和测试基线。
- Bitwarden、1Password 等主流密码管理器公开强调端到端加密、客户端本地解密、主密码不可由服务端恢复、自动锁定、剪贴板清理、用户明确触发填充、透明导出和隐私保护。

因此，本产品的整体方案应明确为：本地保险库是信任中心，用户主动选择是核心交互，安全键盘是主动私密输入的关键入口，平台凭据集成是标准登录场景的兼容增强，云同步是可选增强能力。

## 3. 产品形态

产品由六个体验面组成。

### 3.1 主保险库应用

主应用是用户管理长期数据的地方。它负责创建保险库、解锁保险库、管理条目、组织字段、配置备份迁移，以及在后续版本中管理可选云功能。

主应用应该给用户一种稳定、可信、可审计的感觉。它不追求自动化带来的“神奇感”，而是强调用户知道数据在哪里、发生了什么、下一步会产生什么后果。

主应用的核心任务包括：

- 创建和解锁本地保险库。
- 新建、编辑、搜索和归档条目。
- 管理密码、用户名、TOTP、备注、自定义字段和标签。
- 生成密码和 TOTP 验证码。
- 执行加密导出、导入和恢复。
- 管理生物识别解锁、剪贴板兜底和安全提醒。
- 管理平台凭据集成、自动填充授权和系统设置引导。
- 在后续版本中管理账号、设备、订阅和云同步。

### 3.2 平台凭据集成层

平台凭据集成层是产品兼容标准登录体验的增强层。它负责在平台允许的场景中，把保险库中的登录凭据、Passkey 和 TOTP 交给操作系统认可的凭据选择流程，让用户在兼容的应用和网页中通过系统 UI 完成选择和填充。

这一层不应替代本地保险库，也不应替代安全键盘，更不应让系统或服务端获得保险库明文。它只是一个受系统调用、受用户授权、受最小解锁状态约束的桥接层。

平台凭据集成层的核心任务包括：

- 在 Android 上评估并实现 Credential Manager Provider、Autofill Service 和 IME inline suggestions。
- 在 iOS 上评估并实现 AutoFill Credential Provider Extension，覆盖密码和 TOTP。
- 在 HarmonyOS 上调研等价的凭据、自动填充或输入法扩展能力。
- 当保险库锁定时，通过系统选择入口引导用户解锁。
- 只向系统返回用户选择或系统请求范围内的必要凭据。
- 记录平台不支持或行为不稳定的场景，并提供安全键盘或复制兜底。

### 3.3 安全键盘

安全键盘是用户在其他应用中主动输入私密内容的地方。它不是普通键盘，也不默认提供词典建议。它的主要职责是展示保险库入口、搜索条目、选择字段，并把用户明确选择的值插入当前输入框。

安全键盘不是平台凭据能力的临时替代品，而是主动私密输入的核心入口。它更适合处理非标准字段、地址、备注、自定义私密文本、平台自动填充失败场景，以及用户希望完全手动控制输入的场景。

安全键盘的设计重点是“短、准、克制”。用户打开键盘时通常已经在某个目标输入框中，任务压力高、空间有限、容错空间小。因此键盘不应承载复杂管理功能，也不应鼓励用户在键盘里做大量编辑。

安全键盘的核心任务包括：

- 展示锁定状态，并引导用户完成解锁。
- 搜索条目或打开最近/收藏条目。
- 展示条目中可输入的字段。
- 插入用户明确点击的字段值。
- 对高敏感字段进行必要提醒或重新认证。
- 在平台限制插入时提供复制等兜底方式。

### 3.4 本地保险库

本地保险库是产品的信任中心。它保存所有用户私密数据，并且在本地完成加密。无论用户是否登录、是否联网，本地保险库都必须可创建、可解锁、可编辑、可备份、可迁移。

本地保险库不只是一个数据文件，也是一组用户承诺：用户拥有自己的数据，服务端不是数据恢复的默认依赖，产品不会为了商业功能而阻断本地使用。

本地保险库的核心设计要求包括：

- 主密码是根秘密，忘记后无法由服务端恢复。
- 生物识别是便利机制，不替代主密码。
- 敏感字段必须加密存储，并从日志、分析和崩溃报告中排除。
- 条目标题、标签等元数据也应优先纳入加密载荷。
- 导入、迁移和同步失败不能破坏上一份有效保险库。

### 3.5 备份与迁移

备份与迁移是用户安全感的一部分，不应只作为云服务的附属功能存在。产品必须让用户可以用加密包迁移数据，即使用户不创建账号，也能把保险库带到另一台设备。

备份与迁移体验应该强调结果可验证、失败可恢复、风险可理解。用户需要知道导出的是什么、谁能打开、丢失主密码意味着什么、导入是否会覆盖现有数据。

备份与迁移的核心任务包括：

- 创建加密导出包。
- 从加密包导入或恢复保险库。
- 在本地设备之间传输保险库。
- 校验传输包完整性和来源信息。
- 在导入前展示将要合并、替换或新增的结果。
- 在失败时保留现有保险库并说明原因。

### 3.6 可选云服务

云服务是增强能力，不是基础能力。它可以提供多设备同步、设备管理、版本历史、订阅权益和团队协作，但不能改变本地保险库的离线可用性。

云服务的产品定位必须非常清楚：账号用于服务访问和权益管理，不用于解密用户保险库。即使用户退出登录或同步失败，本地保险库也应该继续可用。

可选云服务的核心任务包括：

- 登录账号。
- 启用或关闭端到端加密云同步。
- 管理已注册设备。
- 查看同步状态、版本历史和冲突。
- 处理订阅权益。
- 后续支持家庭或团队共享。

## 4. 核心体验模型

产品围绕四个高频循环展开。

### 4.1 保存循环

保存循环发生在主应用中。

```text
创建或解锁保险库
  -> 新建条目
  -> 添加字段
  -> 设置敏感性、标签和收藏
  -> 本地加密保存
  -> 可选导出或同步
```

保存循环的设计重点是让用户放心把秘密交给本地保险库。每个关键步骤都应避免暗示服务端可以恢复明文，也不应要求登录作为本地保存前提。

### 4.2 平台填充循环

平台填充循环发生在系统凭据 UI 中。

```text
用户聚焦登录或验证码字段
  -> 系统请求可用凭据
  -> 产品判断保险库锁定状态
  -> 用户解锁或选择凭据
  -> 平台凭据层返回必要字段
  -> 系统完成填充
  -> 产品清理短时授权和敏感值
```

平台填充循环的设计重点是遵守系统能力边界，并服务主动选择模型。产品可以使用平台推荐接口增强登录体验，但仍必须保持用户确认、最小暴露、本地解密和可撤销授权。

### 4.3 键盘输入循环

键盘输入循环发生在安全键盘中。

```text
切换到安全键盘
  -> 解锁或确认短时授权
  -> 搜索或选择条目
  -> 选择具体字段
  -> 插入或复制字段值
  -> 清理短时敏感值
```

键盘输入循环的设计重点是明确选择。键盘可以辅助搜索、排序和展示，但不能替用户判断当前目标一定匹配某个账号，也不能自动填入敏感值。

### 4.4 迁移循环

迁移循环发生在主应用和可选工具中。

```text
选择备份或迁移方式
  -> 生成加密包或同步载荷
  -> 传输到目标位置或设备
  -> 目标端验证包和版本
  -> 用户确认导入策略
  -> 本地解密、合并或恢复
```

迁移循环的设计重点是可控和可恢复。失败导入不能破坏现有数据，冲突必须可见，用户不能被迫使用第一方云完成迁移。

## 5. 信息架构骨架

后续信息架构可以基于以下一级结构展开。

### 5.1 主应用一级区域

| 区域 | 主要用途 | 首批版本要求 |
| --- | --- | --- |
| 保险库 | 查看、搜索、创建和编辑条目。 | v0.2 必须成型。 |
| 生成器 | 生成密码，后续可扩展生成私密字段模板。 | v0.2 必须支持密码生成。 |
| 平台填充 | 系统凭据、自动填充、TOTP 填充和权限引导。 | v0.2 开始评估，v1.0 前应确定首平台策略。 |
| 备份与迁移 | 加密导出、导入、本地传输和恢复。 | v0.2 支持导出/导入，v0.3 强化迁移。 |
| 安全设置 | 主密码、生物识别、自动锁定、剪贴板、安全提醒。 | v0.2 必须成型。 |
| 云与设备 | 登录、同步、设备、版本历史、订阅。 | v1.1 起启用。 |

### 5.2 条目结构

条目是用户理解保险库的核心单位。一个条目代表一个账号、身份、支付信息、安全备注、模板或自定义私密内容。

条目应包含以下体验元素：

- 标题：用户用于识别条目的主名称。
- 类型：登录、安全备注、身份、支付、模板或自定义。
- 字段：用户名、密码、邮箱、电话、TOTP、文本、地址、备注等。
- 敏感性：普通、敏感、高敏感。
- 操作行为：插入、复制、仅显示。
- 组织信息：标签、收藏、归档状态。
- 平台匹配信息：域名、应用包名、App ID、关联域、字段用途和可信绑定状态。

### 5.3 平台凭据集成一级对象

平台凭据集成不一定表现为主应用中的常驻页面，但需要在产品信息架构中有明确对象。

| 对象 | 用途 | 设计约束 |
| --- | --- | --- |
| 凭据提供方状态 | 显示系统是否已启用本产品作为凭据提供方。 | 只做状态和引导，不强迫用户启用。 |
| 自动填充设置 | 引导用户进入系统设置或产品设置。 | 说明平台能力和限制。 |
| 匹配规则 | 管理条目与网站、应用或包名的关系。 | 不承诺识别钓鱼，仅说明匹配依据。 |
| TOTP 填充能力 | 说明验证码是否可通过系统填充。 | 不在日志或同步元数据中保存生成码。 |
| 兜底路径 | 当平台填充不可用时转向键盘、复制或主应用。 | 保持用户可控，不制造失败死路。 |

### 5.4 安全键盘一级区域

安全键盘空间有限，应优先保留以下区域：

| 区域 | 用途 | 设计约束 |
| --- | --- | --- |
| 锁定/解锁区 | 展示锁定状态并触发解锁。 | 不暴露任何条目内容。 |
| 搜索区 | 搜索条目和字段。 | 不上传搜索词，不训练词典。 |
| 条目列表 | 展示最近、收藏或搜索结果。 | 不暗示目标身份真实性。 |
| 字段选择区 | 展示可插入字段。 | 用户点击具体字段后才插入。 |
| 兜底操作区 | 复制、打开主应用、说明平台限制。 | 仅在需要时出现。 |

## 6. 关键状态设计

整体设计必须提前定义几个跨模块状态，避免后续实现各自为政。

### 6.1 未创建保险库

用户首次打开产品时，不应被账号登录拦截。首要任务是创建本地保险库和主密码，并说明恢复边界。

### 6.2 已锁定

锁定状态下，主应用和键盘都不得展示秘密字段。键盘可以展示产品身份和解锁入口，但不应泄露条目标题、最近使用记录或敏感元数据，除非后续安全评审明确允许某些低敏信息展示。

平台凭据入口在锁定状态下可以展示“需要解锁”的认证动作，但不应向系统返回明文凭据。

### 6.3 已解锁

已解锁状态应有明确的生命周期。主应用可以提供完整管理能力；键盘只获得短时、最小授权，用于读取用户选择的字段。

平台凭据层也应采用短时授权。即使主应用已解锁，平台请求仍应按字段敏感性、目标匹配和用户动作决定是否需要再次确认。

### 6.4 离线

离线不是异常状态，而是核心支持状态。产品在离线时仍应允许创建、搜索、编辑、输入、导出和导入本地保险库。云同步入口可以显示不可用或待同步，但不得阻断本地任务。

### 6.5 退出登录

退出登录只影响账号、云同步、设备管理和订阅相关能力。退出登录不得删除或锁死本地保险库，除非用户明确选择删除本地数据。

### 6.6 平台限制

当平台限制键盘插入、共享容器或敏感字段访问时，产品应直接说明限制，并提供复制、打开主应用或手动导入等兜底路径。限制说明不能伪装成用户错误。

平台凭据接口同样可能受系统版本、浏览器支持、应用字段标注、关联域配置和用户系统设置影响。产品应把这些限制作为平台能力状态展示，而不是把失败归因给用户。

## 7. 平台一致性与差异

产品应在三类平台上保持同一套心智模型：本地保险库、主动选择、安全键盘、加密迁移、可选云服务。平台差异只应体现在能力边界、权限说明和交互兜底上。

| 平台 | 一致体验 | 允许差异 |
| --- | --- | --- |
| Android | 主应用、本地加密、Credential Manager/Autofill、安全键盘、主动字段插入。 | v0.1 可先验证键盘，但 v0.2 起应评估凭据提供方和 Autofill Service。 |
| iOS | 主应用、本地保险库、AutoFill Credential Provider、键盘扩展、复制兜底。 | 第三方键盘在敏感字段中的限制必须作为明确约束；密码和 TOTP 应优先评估系统 AutoFill。 |
| HarmonyOS | 主应用、输入法集成、本地加密、主动选择。 | 输入法 API、凭据/自动填充能力、签名、权限和分发渠道需要先完成验证。 |

跨平台体验不追求每个按钮位置完全一致，而追求以下一致性：

- 同一个主密码和保险库概念。
- 同一个“用户明确选择字段后才输入”的规则。
- 同一个“不登录也能本地使用”的承诺。
- 同一个“云端只存密文”的安全边界。
- 同一个“平台限制必须明说并提供兜底”的处理方式。

系统自动填充和凭据管理可以根据平台能力提供更顺滑的登录体验，但产品不能把“系统推荐了某条凭据”表述成“目标一定可信”，也不能把自动填充写成产品主路径。

## 8. 版本层级设计

产品演进应先保证整体闭环，再扩展高级能力。

### 8.1 v0.1：证明主动选择式输入成立，同时调研平台凭据路径

v0.1 不追求完整保险库体验，而是证明一个关键产品假设：用户可以主动选择字段并插入，且过程不需要网络、不收集普通键入文本。安全键盘是验证这一假设的首个实现面。

同时，v0.1 可以并行完成 Android Credential Manager/Autofill、iOS AutoFill Credential Provider 和 HarmonyOS 等价能力的调研。调研目标不是替代主动选择主路径，而是确认哪些平台能力可以作为兼容增强。

### 8.2 v0.2：形成离线产品闭环

v0.2 要让产品成为一个真正可用的离线工具。用户应该能够创建保险库、保存条目、用键盘输入、生成密码和 TOTP、导出和导入保险库。

如果首平台能力验证通过，v0.2 应加入最小可用的系统凭据或自动填充集成；如果暂缓，也必须在文档中说明原因和后续版本入口。

v0.2 的体验完成度比功能数量更重要。只要核心闭环不稳，就不应推进云同步和高级商业能力。

### 8.3 v0.3：增强用户可控迁移

v0.3 重点解决跨设备迁移，而不是直接把用户推向第一方云。加密包、本地配对、导入校验和冲突处理是这一阶段的主要设计对象。

### 8.4 v1.0：公开发布质量

v1.0 要把首个受支持平台的离线体验打磨到发布质量。它需要稳定的本地保险库、稳定的主动输入路径、可用的平台凭据兼容能力、清晰的隐私和安全文档、可靠的备份恢复，以及平台限制说明。

### 8.5 v1.1：可选云同步

v1.1 才进入登录、同步、设备管理、版本历史和订阅权益。云功能必须作为增强层出现，不能反向改变本地能力的产品承诺。

## 9. 商业版本设计边界

商业版本不能破坏核心信任。付费能力可以围绕服务成本、协作效率、高级组织和高级工具展开，但不能把用户的数据出口锁死。

设计边界如下：

- 免费离线版必须保留本地保险库、安全键盘、基础 TOTP、加密导出/导入和手动迁移。
- Plus 可以承载第一方云同步、多设备同步、设备管理、版本历史和云备份恢复。
- Pro 可以承载多保险库、高级模板、安全报告、高级迁移工具和自定义连接器。
- 家庭或团队版可以承载共享保险库、成员权限和审计友好的活动元数据。
- 任何版本都不能让服务端获得保险库明文。
- 帮助用户备份、迁出和恢复控制权的能力，不应完全锁定在第一方云之后。

## 10. 明确不做的体验

为了避免产品心智漂移，以下体验不应进入首批整体设计：

- 不做只依赖系统自动填充、且缺少用户主动控制能力的密码管理器。
- 不做只依赖自定义键盘、却忽略系统凭据接口的密码管理器。
- 不做自动判断当前网站或应用真实性的安全评分。
- 不做默认在线词典或云输入法。
- 不做服务端主密码恢复。
- 不做缺少用户确认的敏感字段自动填入。
- 不把登录作为本地保险库创建、解锁、搜索、编辑、导出或使用的前置条件。

## 11. 后续设计顺序

完成本文档后，建议按以下顺序继续推进，每次只深入一层：

1. 信息架构设计：定义主应用、平台凭据集成和安全键盘的导航、页面层级、对象关系和状态入口。
2. 核心交互设计：定义创建保险库、保存条目、平台填充、安全键盘输入、导出导入这五条关键路径。
3. 平台能力设计：分别验证 Android、iOS、HarmonyOS 的凭据接口、自动填充、键盘、权限、存储和解锁限制。
4. 安全实现设计：细化密钥派生、加密载荷、解锁会话、日志清理和导入回滚。
5. v0.1 原型计划：把整体设计收束为可执行的 Android 键盘可行性原型任务，并附带平台凭据能力调研清单。

下一件最自然的事情是完成“信息架构设计”。它应该仍然保持整体层级，不进入像素级 UI，也不进入平台代码实现。
