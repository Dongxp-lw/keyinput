# v0.1 原型计划

本文档把 v0.1 可行性原型收束为可执行任务，并为每个任务定义验收证据。它承接 [产品决策记录：主动选择式私密输入](product-decision-record.md)、[核心交互设计](interaction-design.md)、[平台能力设计](platform-capability-design.md)，并与 [版本与权益计划](version-plan.md) 的 v0.1 退出标准、[测试计划](../testing/test-plan.md) 的 TP-001 至 TP-007 对齐。

本文档不写最终生产代码，也不锁定加密算法参数。它的目标是定义一个最小、可验证的 Android 原型，用来证明产品的核心假设：用户可以主动选择字段并插入，且过程不需要网络、不收集普通键入文本。

## 1. 原型目标

v0.1 原型只验证一个关键产品假设：

> 用户可以从一个默认锁定的安全键盘中，主动选择保险库条目和字段，并把被选中的值插入到其他应用的输入框中；整个过程不依赖账号、不依赖网络、不收集普通键入文本。

原型必须证明：

- 安全键盘可以注册为 Android 输入法并被用户启用。
- 键盘默认锁定，不在锁定态泄露条目或字段。
- 用户解锁后可以搜索条目、选择字段，并通过 `commitText()` 插入被选中的值。
- 插入在普通文本框、密码框、浏览器和 WebView 中的行为被测试并记录。
- 原型路径不发起任何网络调用。
- 日志不包含被选中的字段值。

原型不需要证明：

- 完整本地保险库加密格式。
- 生物识别解锁。
- 平台自动填充或 Credential Provider。
- 导出/导入或迁移。
- iOS 或 HarmonyOS 的实现。

## 2. 平台与范围

首平台：Android。理由见 [平台能力设计](platform-capability-design.md) 第 7 节：Android IME 能力官方明确，最适合验证主动选择式输入；iOS 键盘限制会干扰原型判断；HarmonyOS 输入法能力仍需独立验证。

范围边界：

- 原型构建 Android 主应用外壳和安全键盘输入法。
- 原型使用最小保险库实现，可以是内存或本地加密模拟，不要求最终文件格式。
- 原型不联网，不登录，不收集普通键入文本。
- iOS 和 HarmonyOS 在 v0.1 只做能力调研记录，不构建实现。

## 3. 调研转译

本轮使用 Android 官方资料校准原型的构建和验证方式。调研用于确定如何验证，而不是改变产品主路径。

### 3.1 跨应用 UI 测试

UI Automator 官方资料说明它可以在被测应用进程之外驱动用户应用和系统应用，按文本或资源 ID 查找元素、点击、`setText`、截图，并通过 `ResultsReporter` 关联测试产物。

对原型的转译：

- 安全键盘插入是跨应用行为，适合用 UI Automator 在浏览器、备忘录等目标应用中验证插入结果。
- 可用 UI Automator 检查目标输入框最终文本是否等于被选中的字段值。
- 可用截图记录密码框、浏览器、WebView 中的实际表现。
- 输入法的系统启用是系统设置操作，需在测试前手动或脚本化完成。

### 3.2 instrumented 测试

Android instrumented 测试运行在真机或模拟器上，使用 `AndroidJUnitRunner`，源码位于 `androidTest`。它能访问框架 API，适合验证依赖真实设备行为的场景。

对原型的转译：

- 键盘插入、字段选择和锁定态行为应优先用 instrumented 测试和 UI Automator 验证。
- 纯逻辑（如条目搜索、字段排序）可用本地单元测试。

### 3.3 离线强约束

Android 应用的网络能力依赖在 manifest 中声明的权限。`<uses-permission>` 用于声明系统权限。网络安全配置可设置 `cleartextTrafficPermitted="false"`，且 Android 9 起明文默认关闭。

对原型的转译：

- 原型不声明 `android.permission.INTERNET`，从根本上移除网络套接字能力。
- 额外配置网络安全配置，关闭明文流量，作为第二层防护。
- 离线保证主要依赖“不声明 INTERNET 权限”，其余手段作为验证补充。

### 3.4 开发期防错

StrictMode 可检测主线程磁盘和网络访问，以及泄漏资源。官方明确说明 StrictMode 是尽力而为的开发工具，不是安全机制，也不一定能捕获 JNI 中的访问。

对原型的转译：

- 开发构建启用 StrictMode，捕获意外的网络或磁盘访问和资源泄漏。
- StrictMode 只作为开发期信号，不作为离线或隐私的最终证明。
- 离线和隐私结论仍以无 INTERNET 权限、网络抓包和日志扫描为准。

## 4. 原型构建块

原型由四个构建块组成。

```text
v0.1 原型
  -> 主应用外壳
  -> 最小保险库
  -> 安全键盘输入法
  -> 验证工具
```

### 4.1 主应用外壳

最小职责：

- 提供首次进入和创建占位保险库的入口。
- 提供少量预置条目用于原型验证。
- 引导用户到系统设置启用安全键盘。
- 不要求登录，不要求网络。

### 4.2 最小保险库

最小职责：

- 保存少量条目和字段，覆盖用户名、密码、自定义字段、备注。
- 提供字段类型、敏感级别和输入行为信息。
- 可以是内存或本地加密模拟实现。
- 不要求最终加密格式或迁移能力。

### 4.3 安全键盘输入法

最小职责：

- 注册为 Android `InputMethodService`。
- 默认显示锁定态。
- 锁定态不展示条目标题、字段标签、最近使用或搜索历史。
- 解锁态提供搜索、条目选择和字段选择。
- 通过 `commitText()` 插入被选中的字段值。
- 提供切换到其他输入法的入口。
- 对密码字段不在键盘 UI 或候选区明文展示秘密。
- 不收集、不上传普通键入文本，不提供在线词库。

### 4.4 验证工具

最小职责：

- instrumented 测试和 UI Automator 脚本验证插入行为。
- 日志扫描脚本检查字段值是否泄漏。
- 离线验证：确认无 INTERNET 权限、网络抓包无流量。
- 开发构建启用 StrictMode。

## 5. 任务分解

每个任务定义目的、交付物和验收证据。

### PROTO-001 项目骨架与离线基线

- 目的：建立 Android 工程骨架，并从根本上保证离线。
- 交付物：主应用模块、键盘模块、不声明 INTERNET 权限、网络安全配置关闭明文、开发构建启用 StrictMode。
- 验收证据：manifest 中无 INTERNET 权限；StrictMode 在开发构建生效；构建可运行。

### PROTO-002 最小保险库与预置数据

- 目的：提供原型验证所需的条目和字段。
- 交付物：最小保险库实现、预置条目（含用户名、密码、自定义字段、备注）、字段类型和敏感级别。
- 验收证据：可读取预置条目；字段携带类型和敏感级别；纯逻辑有本地单元测试。

### PROTO-003 安全键盘注册与锁定态

- 目的：让键盘可被系统启用，并默认安全锁定。
- 交付物：`InputMethodService` 注册、输入法设置入口、锁定态 UI、切换输入法入口。
- 验收证据：键盘可在系统设置启用；锁定态不展示条目标题、字段标签、最近使用、搜索历史；存在切换其他键盘入口。

### PROTO-004 解锁态与主动字段选择

- 目的：实现主动选择式输入的核心交互。
- 交付物：解锁流程、条目搜索、字段列表、字段选择动作。
- 验收证据：解锁后可搜索并选择条目；字段列表显示标签和类型，不默认展示密码明文；只有用户点击字段才触发插入。

### PROTO-005 字段插入与跨应用验证

- 目的：验证 `commitText()` 把被选中的值插入目标输入框。
- 交付物：字段插入实现、UI Automator 跨应用验证脚本、密码框/浏览器/WebView 行为记录。
- 验收证据：普通文本框只插入被选中值；密码框插入成功或限制被记录；浏览器和 WebView 行为有测试和截图记录。

### PROTO-006 隐私与离线验证

- 目的：证明原型不收集键入文本、不联网、不泄漏秘密到日志。
- 交付物：日志扫描脚本、网络抓包验证、键入文本不收集的代码与测试确认。
- 验收证据：网络关闭和无 INTERNET 权限下插入路径可用；抓包无流量；日志不含字段值；键盘不收集普通键入文本。

### PROTO-007 平台限制调研记录

- 目的：并行记录 iOS 和 HarmonyOS 平台限制，不构建实现。
- 交付物：iOS 键盘扩展受限场景记录、HarmonyOS 输入法能力待验证清单。
- 验收证据：iOS 安全字段替换等限制被记录；HarmonyOS 输入法注册和提交文本能力的验证项被列出。

## 7. 退出标准

v0.1 原型完成需同时满足版本计划退出标准和原型验证：

- 键盘可以把用户选择的值插入常见 Android 文本字段。
- 密码字段和浏览器字段已完成测试并形成文档。
- 原型路径不需要任何网络调用。
- 锁定态不泄露条目和字段。
- 日志不包含被选中的字段值。
- 键盘不收集普通键入文本。
- iOS 和 HarmonyOS 平台限制已记录。

## 8. 风险与兜底

| 风险 | 影响 | 兜底 |
| --- | --- | --- |
| 密码框限制键盘插入 | 部分场景无法插入密码。 | 记录限制，提供复制或打开主应用兜底，并在文档说明。 |
| WebView 行为不一致 | 插入结果不稳定。 | 按目标分别记录，不夸大覆盖范围。 |
| IME 生命周期清退解锁状态 | 用户频繁解锁。 | 使用短时授权，记录体验问题，留待 v0.2 调优。 |
| StrictMode 漏报网络 | 离线结论被高估。 | 离线以无 INTERNET 权限和抓包为准。 |
| HarmonyOS 能力未验证 | 不能承诺时间表。 | 仅记录验证项，不进入版本承诺。 |

## 9. 不在 v0.1 范围

- 最终保险库加密格式和迁移。
- 生物识别和设备凭据解锁。
- 平台自动填充、Credential Provider、TOTP 自动填充。
- 加密导出/导入和跨设备迁移。
- 云账号、同步和订阅。
- iOS 和 HarmonyOS 的实现构建。

## 10. 下一步

v0.1 原型完成并满足退出标准后，进入 v0.2 离线 MVP。下一份文档建议是“v0.2 离线 MVP 计划”，把本地保险库加密格式、主密码、生物识别解锁、密码生成、基础 TOTP、加密导出/导入和剪贴板兜底拆成可执行任务。
