# 产品决策记录：主动选择式私密输入

本文档记录 Private Input Vault 的核心产品决策。后续任何外部调研、平台能力接入、商业版本设计和技术方案，都必须先在这个背景下评估。

## 1. 我们要做什么产品

Private Input Vault 要做的是一个离线优先、用户主动选择的私密输入键盘与加密保险库。

它不是传统意义上以自动填充为中心的密码管理器，也不是普通输入法。它管理的不只是登录密码，还包括用户名、TOTP、地址、身份字段、恢复码、备注、自定义私密文本，以及用户希望反复输入但不希望交给普通键盘或云输入法的内容。

产品的核心承诺是：

- 用户拥有本地保险库，不登录、不联网也能创建、解锁、编辑、输入、导出和迁移。
- 用户主动选择要输入的条目和字段，产品不替用户猜测目标是否可信。
- 产品不承诺钓鱼检测，不判断网站或应用身份真实性。
- 云服务只作为可选增强能力存在，不能成为本地使用前提。
- 服务端不能接触保险库明文、主密码、TOTP 种子或内容加密密钥。

## 2. 为什么需要用户主动选择

这里的“手动”不是要求用户逐字符输入密码，而是要求用户明确选择要输入的私密内容。

这个选择是产品安全边界的一部分，原因如下：

- 自动填充依赖上下文判断，但移动端上下文并不总是可靠。应用、网页、WebView、浏览器、嵌入式登录页、伪装页面和特殊输入框都可能让“应该填哪一条”变得不清楚。
- 产品明确不做钓鱼检测，也不判断目标身份真实性。既然不做这个判断，就不能把“自动选择并填入”作为核心承诺。
- 保险库里有大量非登录字段，例如备注、地址、恢复码、自定义字段和私密文本。这些内容没有稳定的平台自动填充语义，更适合由用户主动选择。
- 对高敏感数据来说，用户看到条目和字段并点击具体值，本身就是一次确认动作。它让用户意识到自己正在把哪个秘密交给哪个输入框。
- 主动选择模型更适合离线优先产品。它不要求账号体系、云同步或平台绑定先成立，产品可以先作为本地工具完整运行。

因此，用户主动选择不是临时兜底，而是产品心智、隐私边界和责任边界的核心。

## 3. 平台能力的位置

Android Credential Manager/Autofill、iOS AutoFill Credential Provider、IME inline suggestions 等平台能力值得调研和接入，但它们是兼容层和增强层，不是产品主心智。

平台能力适合解决这些问题：

- 在标准登录和验证码场景中减少用户操作成本。
- 进入系统认可的凭据选择流程，提高与应用和浏览器的兼容性。
- 在用户已经明确启用系统凭据提供方时，提供更自然的登录体验。
- 在平台允许的范围内承载密码、Passkey 和 TOTP 的标准填充。

平台能力不能改变这些边界：

- 不能替代本地保险库。
- 不能要求登录后才能使用本地能力。
- 不能让系统推荐被表述为“目标一定可信”。
- 不能要求服务端持有或恢复保险库明文。
- 不能覆盖所有私密输入场景，因此不能取代安全键盘。

## 4. 如何判断“更好的方案”

外部调研发现的更好方案，只有在不破坏上述产品背景时才应采纳。

评估一个新方案时，必须先回答：

- 它是否保持用户主动选择，而不是让产品替用户判断目标可信？
- 它是否保持不登录、不联网也能使用本地保险库？
- 它是否覆盖非登录私密字段，而不只服务账号密码？
- 它是否避免服务端接触保险库明文和主密码材料？
- 它是否能在平台失败或受限时提供清楚、可控的兜底路径？
- 它是否诚实说明平台能力边界，而不是把限制伪装成用户错误？

如果答案是否定的，即使它是主流方案，也只能作为参考，不能成为产品主路径。

## 5. 后续文档约束

后续所有产品和技术文档应遵守以下写法：

- 先说明产品为什么坚持主动选择，再讨论平台能力如何增强体验。
- 避免把“平台推荐路径”“行业主流做法”直接写成产品目标。
- 避免把安全键盘写成临时兜底；它是主动私密输入的核心入口之一。
- 避免把自动填充写成主路径；它是标准登录场景的兼容增强。
- 如果引入更优方案，必须说明它如何服务离线优先、主动选择和本地加密保险库。

一句话基线：主流方案可以帮助产品更好，但不能替产品决定它是谁。
