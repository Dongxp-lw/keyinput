# v1.0 公开发布计划

本文档把 v1.0 公开发布拆成可执行任务，并为每个任务定义验收证据。它把 [v0.2 离线 MVP 计划](v0.2-mvp-plan.md) 和 [v0.3 跨设备迁移计划](v0.3-migration-plan.md) 的成果推进到生产质量，复用 [安全实现设计](../technical/security-implementation-design.md) 的密码学决策，并对齐 [版本与权益计划](version-plan.md) 的 v1.0 退出标准、[路线图](roadmap.md)、[平台能力设计](platform-capability-design.md) 与 [测试计划](../testing/test-plan.md)。

本文档服从 [产品决策记录：主动选择式私密输入](product-decision-record.md)：主动选择是核心心智模型；生物识别是便利机制，不替代主密码；平台凭据与自动填充是兼容增强，不是产品主路径。

## 1. 目标

v1.0 在首个受支持平台 Android 上交付稳定的离线优先产品，并清楚说明平台特定限制。它要让用户能够：

> 在生产质量的本地保险库和安全键盘上，离线创建、使用、备份和恢复保险库，使用主密码和生物识别解锁，主动选择并插入字段，使用基础 TOTP；同时产品通过发布安全质量门禁，平台限制在产品内和发布说明中清楚记录，核心本地使用不需要登录。

## 2. 平台与范围

首平台为 Android。iOS 和 HarmonyOS 在 v1.0 不构建实现，平台限制继续记录。

v1.0 在范围内：

- 生产级本地保险库。
- 生产级安全键盘。
- 基础 TOTP。
- 加密备份与恢复。
- 清晰的隐私与安全文档。
- 发布测试套件与手动 QA 清单。
- 发布安全质量门禁。

v1.0 不在范围内：

- 云账号、云同步、订阅（属于 v1.1）。
- 多保险库与高级模板（属于 Pro）。
- iOS 与 HarmonyOS 实现构建。
- v0.1 至 v0.3 之外的新功能。

## 3. 调研转译

本轮以 OWASP MASVS 和 Google Play 发布要求校准发布质量门禁与平台前置；密码学与卫生决策复用安全实现设计。

### 3.1 发布安全质量门禁（OWASP MASVS）

OWASP MASVS 把移动应用安全划分为八个控制分组，适合作为发布安全质量门禁的结构：

- MASVS-STORAGE：敏感数据的安全存储（静态数据）。
- MASVS-CRYPTO：保护敏感数据的密码学功能。
- MASVS-AUTH：认证与授权机制。
- MASVS-NETWORK：安全的网络通信（传输中数据）。
- MASVS-PLATFORM：与平台和其他应用的安全交互。
- MASVS-CODE：数据处理最佳实践与保持应用更新。
- MASVS-RESILIENCE：抵抗逆向与篡改的韧性。
- MASVS-PRIVACY：保护用户隐私的控制。

### 3.2 平台发布前置（Google Play target API level）

- 据 Google Play 官方说明，新应用与应用更新需面向较新的 Android 版本（页面所列为 Android 15 / API 35 起，部分品类要求略低）。
- Play 通常每年提高该要求，具体目标 API 级别以发布时的 Play 政策为准，待确认。
- 发布前置还包括正式应用签名、生产构建配置和依赖合规。

### 3.3 复用的安全决策

- 保险库加密、KDF、密钥层级、文件格式、解锁会话、内存与日志卫生：见 [安全实现设计](../technical/security-implementation-design.md)。
- TOTP 行为与测试向量：见 [v0.2 离线 MVP 计划](v0.2-mvp-plan.md)。
- 迁移与备份恢复的密码学规则：见安全实现设计第 8 节与 [v0.3 跨设备迁移计划](v0.3-migration-plan.md)。

## 4. 发布安全质量门禁

发布前必须通过以下 MASVS 对齐的门禁。每个分组结合本产品给出检查要点。

| MASVS 分组 | 发布检查要点 |
| --- | --- |
| MASVS-STORAGE | 保险库内容加密落盘；秘密不进日志；敏感数据排除系统备份；仅在应用沙箱内加密存储。 |
| MASVS-CRYPTO | 按安全实现设计使用 Argon2id 与 XChaCha20-Poly1305；CSPRNG；无硬编码密钥；密钥用途隔离。 |
| MASVS-AUTH | 主密码为根秘密；生物识别绑定平台密钥库；新生物识别注册使密钥失效；高敏感操作可重新认证。 |
| MASVS-NETWORK | v1.0 不声明 INTERNET 权限；关闭明文流量；确认没有隐藏网络面。 |
| MASVS-PLATFORM | 键盘不缓存敏感输入；以 FLAG_SECURE 防截屏；通知不泄密；剪贴板自动清除；最小化导出组件。 |
| MASVS-CODE | 面向较新平台 API；依赖无已知漏洞并产出 SBOM；对导入数据做输入校验。 |
| MASVS-RESILIENCE | 正式签名；关闭 debuggable；生产构建移除调试日志；混淆与完整性校验按威胁模型决定，属防御纵深，部分可选。 |
| MASVS-PRIVACY | 最小权限；数据收集声明准确（默认不收集、不联网）；不使用追踪标识。 |

## 6. 任务分解

每个任务定义目的、交付物和验收证据。

### REL-001 生产级保险库与键盘加固

- 目的：把 MVP 的保险库与键盘推进到生产质量。
- 交付物：稳定性与边界条件处理、敏感界面 FLAG_SECURE、键盘不缓存敏感输入、错误处理不泄密。
- 验收证据：核心路径稳定通过回归；敏感界面防截屏生效；键盘缓存检查通过；错误信息不含秘密。

### REL-002 基础 TOTP 生产化

- 目的：把基础 TOTP 推进到生产质量。
- 交付物：稳定的 TOTP 生成、种子加密存储、刷新与时钟漂移提示。
- 验收证据：RFC 6238 测试向量持续通过；种子不记日志；时钟漂移有可理解提示。

### REL-003 加密备份与恢复

- 目的：提供可靠的本地备份与恢复。
- 交付物：加密备份生成、恢复校验、失败回滚。
- 验收证据：备份不可明文读取；恢复内容与源一致；被篡改备份被拒绝；恢复失败不破坏现有保险库。

### REL-004 隐私与安全文档

- 目的：清楚说明产品的隐私与安全立场和边界。
- 交付物：隐私说明、安全说明、主密码恢复边界、平台限制说明、数据收集声明（默认不收集、不联网）。
- 验收证据：文档覆盖恢复边界与平台限制；数据收集声明与实际行为一致；不夸大安全承诺。

### REL-005 发布安全质量门禁

- 目的：以 MASVS 对齐的门禁把关发布。
- 交付物：按第 4 节八个分组的检查清单与执行记录。
- 验收证据：每个分组检查项有明确结论；阻断项清零；可选项（如部分 RESILIENCE 控制）有按威胁模型的取舍说明。

### REL-006 发布测试套件与手动 QA 清单

- 目的：形成可重复的发布验证。
- 交付物：聚合 v0.1 至 v0.3 测试的回归套件、发布专项检查、手动 QA 清单。
- 验收证据：回归套件可重复执行并通过；手动 QA 清单覆盖关键路径；缺陷处理规则中的阻断项全部清零。

### REL-007 发布工程

- 目的：满足平台发布前置并保证构建可信。
- 交付物：正式应用签名、面向较新平台 API、关闭 debuggable、移除生产调试日志、可复现构建。
- 验收证据：目标 API 满足发布时 Play 要求；签名有效；生产构建不含调试日志；构建可复现。

### REL-008 平台限制记录

- 目的：在产品内和发布说明中清楚记录平台限制。
- 交付物：产品内限制提示、发布说明中的平台限制章节、iOS 与 HarmonyOS 现状记录。
- 验收证据：产品内与发布说明的限制描述一致；限制以正常状态呈现而非异常失败。

## 7. 测试与 QA

- 发布回归套件聚合测试计划的 v0.1（TP-001..007）、v0.2（TP-101..109）、v0.3（TP-301..308）。
- 在此基础上增加发布专项检查与手动 QA 清单，覆盖加固、备份恢复、门禁验证。
- 建议在测试计划新增 v1.0 发布回归与门禁小节，沉淀发布专项用例。
- 缺陷处理沿用测试计划：明文秘密入日志、保险库损坏、本地使用要求登录、隐藏网络依赖均为发布阻断项。

## 8. 退出标准

v1.0 完成需同时满足版本计划退出标准和发布门禁：

- 发布质量门禁通过，MASVS 八个分组无阻断项。
- 平台限制已在产品内和发布说明中记录。
- 核心本地使用不需要登录。
- 保险库、键盘、TOTP、备份恢复达到生产质量。
- 发布回归套件与手动 QA 清单通过。

## 9. 风险与兜底

| 风险 | 影响 | 兜底 |
| --- | --- | --- |
| Play 目标 API 要求逐年提高 | 发布被阻。 | 以发布时 Play 政策为准确定目标 API，提前适配。 |
| 过度追求 RESILIENCE 控制 | 投入与威胁模型不匹配。 | 按威胁模型取舍，记录可选项的决定。 |
| 平台键盘限制影响部分场景 | 个别插入不可用。 | 提供剪贴板兜底并在限制说明中记录。 |
| 文档夸大安全承诺 | 误导用户、违反诚实原则。 | 明确恢复边界与平台限制，不承诺钓鱼检测。 |
| 依赖存在已知漏洞 | 发布带病。 | SBOM/SCA 扫描并在门禁中清零阻断项。 |

## 10. 不在 v1.0 范围

- 云账号、云同步、订阅（属于 v1.1）。
- 多保险库与高级模板。
- 平台自动填充、Credential Provider、TOTP 自动填充。
- iOS 与 HarmonyOS 实现构建。

## 11. 下一步

v1.0 发布并满足退出标准后，进入 v1.1 可选云同步。下一份文档建议是“v1.1 云同步计划”，把账号身份、端到端加密云同步、设备管理、同步冲突处理、版本历史和订阅权益拆成可执行任务与验收证据，并保持本地使用永不依赖云的承诺。建议同时在测试计划补充 v1.0 发布回归小节。
