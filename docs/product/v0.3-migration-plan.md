# v0.3 跨设备迁移计划

本文档把 v0.3 跨设备迁移拆成可执行任务，并为每个任务定义验收证据。它承接 [v0.2 离线 MVP 计划](v0.2-mvp-plan.md) 的加密导出/导入，复用 [安全实现设计](../technical/security-implementation-design.md) 第 8 节的迁移包密码学规则与 [数据模型](../technical/data-model.md) 的 TransferPackage，并对齐 [版本与权益计划](version-plan.md) 的 v0.3 退出标准、[路线图](roadmap.md) 阶段 2，以及 [核心交互设计](interaction-design.md) 的可恢复与防错原则。

本文档服从 [产品决策记录：主动选择式私密输入](product-decision-record.md)：迁移由用户自主控制、不需要第一方云账号；主动选择仍是核心心智模型；平台凭据与自动填充是兼容增强，不是产品主路径。

## 1. 目标

v0.3 让用户在不依赖第一方云的前提下，把本地保险库迁移到另一台设备。在 v0.2 已交付文件级加密导出/导入之后，v0.3 要让用户能够：

> 在两台设备之间，通过加密迁移包和用户自主控制的本地通道完成迁移，迁移前可校验和预览，导入时能处理与现有保险库的冲突，且导入失败不会破坏现有保险库。

## 2. 平台与范围

首平台延续 Android。iOS 和 HarmonyOS 在 v0.3 不构建实现，平台限制继续记录。

v0.3 在范围内：

- 加密迁移包格式与认证保护。
- 设备配对：二维码或手动配对设计。
- 用户自主控制的本地传输通道。
- 迁移校验与导入预览。
- 导入保险库版本与内容的冲突处理。
- 可恢复导入与失败回滚。
- 桌面或 CLI 打包辅助工具的评估。

v0.3 不在范围内：

- 第一方云账号、云同步、订阅（属于 v1.1）。
- 多保险库与高级迁移工具（属于 Pro）。
- iOS 与 HarmonyOS 实现构建。

## 3. 调研转译

本轮在二维码迁移上做了容量调研；迁移包加密、回滚、架构兼容规则直接复用安全实现设计与数据模型。

### 3.1 二维码容量与配对

- 按 QR Code 标准，单个二维码字节模式容量上限约为数 KB（最高版本、最低纠错级别时约 2953 字节），并随纠错级别提高而下降。
- 纠错级别有 L、M、Q、H 四档，级别越高抗污损越强但容量越小，常用 M 约 15%。
- 设计结论：单个二维码一般无法可靠承载完整保险库，因此二维码用于承载短的配对密钥或引导信息，加密载荷通过文件或本地通道传输。
- 配对密钥短小，可使用较高纠错级别提升扫描可靠性。

### 3.2 复用的迁移密码学规则

- 迁移包使用独立于本地内容密钥的域分离迁移密钥，带认证标签，先校验后使用：见 [安全实现设计](../technical/security-implementation-design.md) 第 8 节。
- 导入失败不得覆盖上一份有效保险库，必须可回滚。
- 导入前校验格式与架构兼容性，尽量保留未知字段，避免跨版本丢数据：见数据模型架构迁移规则。

### 3.3 复用的交互原则

- 迁移和导入必须先预览，不直接覆盖本地保险库。
- 迁移中断可重新开始或继续校验。
- 错误提示保留用户上下文，不泄露秘密：见 [核心交互设计](interaction-design.md) 第 3 节。

## 4. 迁移模型

v0.2 已交付文件级加密导出/导入，可用于同设备备份恢复或用户手动搬运文件。v0.3 在此基础上增加设备到设备迁移。

```text
迁移通道
  -> 通道 A：加密迁移包文件（复用 v0.2 导出/导入）
  -> 通道 B：本地设备配对 + 直接传输
        -> 二维码或手动配对建立短时配对密钥
        -> 加密迁移包通过本地通道传输
  公共步骤
  -> 接收端校验迁移包
  -> 接收端预览内容与冲突
  -> 用户选择替换/合并/保留
  -> 可恢复导入，失败回滚
```

要点：

- 两种通道都使用同一种加密迁移包格式。
- 二维码承载配对密钥或引导信息，不承载完整加密载荷。
- 接收端在应用变更前始终先校验、再预览、后导入。

## 5. 迁移构建块

```text
v0.3 跨设备迁移
  -> 迁移包核心
  -> 设备配对
  -> 本地传输通道
  -> 校验与冲突处理
  -> 辅助工具评估
  -> 验证工具
```

- 迁移包核心：迁移包格式、迁移密钥派生、认证加密、版本与架构信息。
- 设备配对：二维码配对密钥生成与扫描、手动配对兜底。
- 本地传输通道：用户自主控制的加密载荷传输，不依赖云。
- 校验与冲突处理：完整性校验、内容预览、替换/合并/保留决策。
- 辅助工具评估：桌面或 CLI 打包工具的可行性评估。
- 验证工具：单元测试、instrumented 测试、离线验证、日志扫描。

## 6. 任务分解

每个任务定义目的、交付物和验收证据。

### MIG-001 迁移包格式与加密

- 目的：定义可在设备间安全传递的迁移包。
- 交付物：按数据模型的 TransferPackage 实现，使用域分离迁移密钥、认证加密、packageVersion 与 cryptoProfile。
- 验收证据：迁移包不可明文读取；篡改任意字节导致校验失败；迁移密钥独立于本地内容密钥；已知答案测试覆盖加密与认证。

### MIG-002 设备配对

- 目的：在两台设备间建立用户确认的配对。
- 交付物：二维码承载短时配对密钥或引导信息、扫描流程、手动配对兜底、配对短时有效。
- 验收证据：可通过二维码或手动方式完成配对；二维码不承载完整加密载荷；配对信息短时失效；配对需要用户在两端确认。

### MIG-003 本地传输通道

- 目的：在不依赖云的前提下传输加密迁移包。
- 交付物：用户自主控制的本地传输实现、传输中只传密文、传输不需要登录与第一方云。
- 验收证据：迁移包以密文传输；传输路径不需要账号或第一方云；传输中断可重试。

### MIG-004 迁移校验与预览

- 目的：在导入前让用户确认将要发生什么。
- 交付物：完整性校验、条目与字段计数、架构兼容性检查、导入预览。
- 验收证据：校验失败的迁移包被拒绝；预览展示将导入的条目数量和冲突；架构不兼容时给出可理解说明。

### MIG-005 导入冲突处理

- 目的：处理接收端已存在保险库的情况。
- 交付物：替换、合并、保留两份的选择，冲突项预览，默认不覆盖。
- 验收证据：默认不静默覆盖现有保险库；用户可选择替换、合并或保留；合并保留双方数据且无静默丢失。

### MIG-006 可恢复导入与回滚

- 目的：保证导入失败不破坏现有数据。
- 交付物：导入事务化、失败回滚、可重新开始或继续校验。
- 验收证据：导入失败后现有保险库保持有效；中断后可重试；回滚不留下损坏的中间状态。

### MIG-007 桌面或 CLI 辅助工具评估

- 目的：评估打包辅助工具的可行性，不强制实现。
- 交付物：桌面或 CLI 工具在迁移包生成与校验中的角色评估、安全边界记录。
- 验收证据：形成评估结论；若纳入，工具不削弱加密与认证保证，也不引入云依赖。

### MIG-008 离线与隐私验证

- 目的：保持离线与隐私保证。
- 交付物：不声明 INTERNET 权限、本地通道验证、日志扫描。
- 验收证据：迁移路径不需要第一方云；日志不含主密码、密钥、字段值、TOTP 种子和迁移密钥。

## 8. 退出标准

v0.3 完成需同时满足版本计划退出标准和迁移验证：

- 用户无需第一方云即可把保险库迁移到另一台设备。
- 迁移包已加密并通过认证保护。
- 导入失败不会破坏现有保险库。
- 二维码或手动配对可建立用户确认的配对。
- 导入前可校验和预览，冲突可由用户决策。

## 9. 风险与兜底

| 风险 | 影响 | 兜底 |
| --- | --- | --- |
| 二维码容量不足以承载完整保险库 | 迁移失败或体验差。 | 二维码只承载配对密钥，载荷走文件或本地通道。 |
| 本地传输通道在部分环境受限 | 设备到设备迁移不可用。 | 回退到 v0.2 文件迁移包路径。 |
| 合并冲突设计不当导致数据丢失 | 用户数据丢失。 | 默认不覆盖、先预览、保留双方、可回滚。 |
| 架构版本不兼容 | 导入失败。 | 校验兼容性、保留未知字段、给出可理解说明。 |
| 桌面/CLI 工具引入云依赖或弱化加密 | 削弱安全保证。 | 仅在不削弱加密与认证、不引入云依赖时纳入。 |

## 10. 不在 v0.3 范围

- 第一方云账号、云同步、订阅（属于 v1.1）。
- 多设备实时同步与冲突合并服务。
- 多保险库与高级迁移工具。
- iOS 与 HarmonyOS 实现构建。

## 11. 下一步

v0.3 完成并满足退出标准后，进入 v1.0 公开发布。下一份文档建议是“v1.0 公开发布计划”，把生产级保险库与键盘、发布质量门禁、平台限制说明和发布 QA 清单拆成可执行任务与验收证据。建议同时在测试计划补充 v0.3 迁移小节。
