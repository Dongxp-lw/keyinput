# 实现文档层总览

本文件定义实现文档层的目标、技术栈策略、统一模板、AI 任务拆分规范和功能顺序。它是 [文档地图与串联](../README.md) 中“实现文档层”的入口，服从 [产品决策记录](../product/product-decision-record.md) 基线。

实现文档把版本计划任务（PROTO/MVP/MIG/REL/SYNC）落到“怎么做”，并在每份文档内附带可由 AI 逐步生成的原子任务。

## 1. 目标与范围

- 为每个小功能提供可实现的how-to：接口、数据结构、步骤、边界与错误处理。
- 在每份实现文档内附带 AI 任务拆分，作为后续由 AI 生成代码的输入。
- 当前阶段仍是写文档，不写最终生产代码。
- 实现文档必须可追溯回设计与技术文档，并映射到测试。

## 2. 技术栈与平台策略

- 平台优先级：先实现 Android；iOS 与 HarmonyOS 的**适配**放到 v1 阶段末尾。
- 架构（已锁定 D-006）：**一套 Rust 共享核心 + 各端原生 UI**，详见 [模块架构与层间契约](../technical/module-architecture.md)。
  - 核心层（**Rust**，现在就建）：保险库格式、加密、序列化、领域逻辑、TOTP、导出导入、迁移/同步逻辑——即 VAULT、ENTRY、GEN、TOTP、IMEX，以及 LOCK 的密钥/会话逻辑、MIGR/SYNC 的逻辑。
  - 原生层（**Kotlin/Android**）：键盘 IME、生物识别、剪贴板、UI、自动锁定生命周期、文件/网络 IO——即 KBD、BIO、CLIP，以及 LOCK/MIGR/SYNC 的平台部分。
- 加密库（已锁定 D-004）：核心加密用 Rust crate（RustCrypto 或 libsodium-sys），**不用** lazysodium-android。
- **重要**：各功能文档第 4 节的 Kotlin/Java 接口与 API 是**逻辑示意**；核心层功能实际以 Rust 实现，权威契约见模块架构。原生层功能确为 Kotlin。
- 密码学选型遵循 [安全实现设计](../technical/security-implementation-design.md)，不在实现文档中另立选型。

## 3. 实现文档模板

每份功能实现文档放在 `docs/implementation/` 下，命名用功能短名，结构固定如下。

```text
# <功能名> 实现文档

1. 范围与目标
2. 依赖的设计与技术决策（链接到上层文档）
3. 平台与技术栈（Android 优先）
4. 接口与数据结构
5. 实现步骤
6. 边界条件与错误处理
7. 安全与隐私要求（引用安全实现设计与 MASVS 相关项）
8. 测试映射（关联 TP 与版本计划任务）
9. AI 任务拆分（见第 4 节规范）
10. 待验证与不在范围
```

要求：

- 第 2 节必须链接到对应的设计与技术文档，不重复其结论。
- 第 7 节必须包含不记录秘密、最小暴露、离线优先等基线要求。
- 第 8 节必须给出可验证的测试关联。

## 4. AI 任务拆分规范

每份实现文档的第 9 节用统一格式列出原子任务，供 AI 逐步生成。任务 ID 用功能短码加序号，例如键盘用 `KBD-01`。

每个原子任务包含：

- 任务 ID 与所属功能。
- 输入：依赖的文档与前置任务。
- 产出物：要生成的文件或代码单元。
- 约束：必须遵守的基线与安全要求。
- 验收证据：如何判断完成。
- 依赖与顺序：可并行或必须串行。

任务粒度要求：

- 每个任务足够小，可由一次 AI 生成完成并独立验证。
- 任务自带验收证据，不依赖隐含上下文。
- 任务显式声明对版本计划任务（如 MVP-005）和测试（如 TP-001）的映射。

## 6. 通用约束

- 服从产品决策记录基线：主动选择是核心，离线优先，平台凭据与自动填充是兼容增强不是主路径。
- 满足安全实现设计的密码学与卫生要求，发布前对齐 [v1.0 发布计划](../product/v1.0-release-plan.md) 的 MASVS 门禁。
- 不记录主密码、密钥、字段值、TOTP 种子和生成的验证码。
- 平台能力未经验证处标注为待验证。

## 7. 文档约定与验证

- 文档使用中文编写，保留必要英文技术术语。
- 新增实现文档后，更新第 5 节状态、[文档地图](../README.md) 和顶层 README 索引。
- 每份文档通过诊断、尾随空白、链接和基线措辞检查。
