# 安全模型

## 1. 安全目标

- 在设备存储被检查时保护保险库内容。
- 避免服务端访问保险库明文。
- 避免收集用户键盘输入。
- 插入敏感值之前要求用户明确操作。
- 保持本地使用不依赖登录或网络。
- 当主密码丢失时，清楚说明恢复边界。

## 2. 信任边界

### 可信

- 用户设备操作系统提供的安全原语。
- 用户批准的生物识别或设备凭据 API。
- 本项目发布的本地应用和键盘代码。
- 经审查和测试后的共享核心实现。

### 不可信任其接触明文

- 第一方云服务。
- 用户选择的云存储提供方。
- 网络传输链路。
- 日志、分析、崩溃报告和支持渠道。

## 3. 数据分级

| 数据 | 分级 | 说明 |
| --- | --- | --- |
| 主密码 | 秘密 | 绝不存储或传输。 |
| 保险库加密密钥 | 秘密 | 只按设计以包装或派生形式保存。 |
| 密码字段 | 秘密 | 静态存储时加密，且绝不写入日志。 |
| TOTP 种子 | 秘密 | 作为高敏感字段处理。 |
| 生成的 TOTP 验证码 | 敏感临时值 | 生命周期短，绝不写入日志。 |
| 条目标题和标签 | 敏感元数据 | 优先在保险库载荷内加密。 |
| 同步对象版本 ID | 元数据 | 可存在于服务端，但不得暴露内容。 |
| 云账号邮箱 | 个人数据 | 仅登录/云功能需要。 |

## 4. 保险库加密方向

最终算法选择必须在实现评审中确定。初始设计方向如下：

- 使用内存困难型密码 KDF，从主密码派生密钥。
- 对保险库载荷使用认证加密。
- 加密所有有意义的保险库内容，包括标题、标签、自定义字段、密码和 TOTP 种子。
- 在保险库头部包含架构版本和密码学参数。
- 对影响解密或迁移的头部和元数据进行认证保护。
- 对保险库载荷、导出包和同步载荷使用不同密钥，或使用域分离。

## 5. 解锁模型

- 主密码是用户的根秘密。
- 生物识别解锁是便利机制，而不是主密码的替代品。
- 启用生物识别时，平台安全存储可以包装本地解锁密钥。
- 应用必须允许用户关闭生物识别解锁。
- 键盘只应获得读取所选字段所需的最小会话授权。

## 6. 键盘隐私规则

- 不收集用户输入文本。
- 不上传用户输入文本。
- 不使用默认在线词库。
- 不基于敏感输入训练本地词库。
- 不记录被选中的字段值。
- 在平台 API 支持时，避免敏感字段暴露在截图中。
- 插入后在可行时尽快从内存中清除被选中的敏感值。

## 7. 云安全规则

- 云同步是可选功能。
- 登录用于账号、权益、设备管理和云对象访问。
- 服务端只存储加密保险库载荷。
- 服务端不得接收主密码或保险库明文。
- 云账号密码重置不得解密保险库数据。
- 恢复机制必须单独设计，并在实现前完成评审。

## 8. 钓鱼与目标身份边界

产品不承诺钓鱼检测，也不会自动判断当前应用或网站应该使用哪一组凭据。用户从键盘或应用中主动选择值。

产品仍应避免误导行为：

- 不基于弱上下文静默推荐凭据。
- 不声称目标输入框是安全的。
- 在插入高敏感值前展示可选提醒。
- 允许用户将条目固定为收藏，但不暗示目标身份真实可信。

## 9. 2FA/TOTP 策略

- TOTP 种子作为高敏感字段支持。
- 把密码和 TOTP 种子存放在同一保险库中很方便，但会降低因子分离程度。
- 产品应诚实说明这一点。
- 后续可以为 TOTP 字段提供更严格的安全模式。

## 10. 日志与遥测策略

初始产品版本默认应不启用遥测。

如果后续增加诊断能力：

- 在可行时必须采用用户选择加入。
- 必须排除秘密和加密保险库载荷。
- 必须形成文档。
- 崩溃报告必须清理字段值和保险库标识符。
