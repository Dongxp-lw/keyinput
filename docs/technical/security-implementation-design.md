# 安全实现设计

本文档把 [安全模型](security-model.md) 的方向性决策落为可实现规格，并与 [数据模型](data-model.md)、[技术架构](architecture.md) 对齐。它服从 [产品决策记录：主动选择式私密输入](../product/product-decision-record.md)：主动选择是安全边界；平台密钥库和生物识别是硬件绑定与便利增强，不替代主密码这一根秘密；平台凭据与自动填充仍是标准登录场景的兼容增强，不是产品主路径。

本文档给出选型、密钥层级、文件格式、解锁会话和卫生规则，作为核心实现评审的输入。最终算法库选择和移动端参数仍需基准测试和实现评审确认，文中以“待评审”“待基准测试”标注。

## 1. 设计依据与调研转译

本轮以权威公开资料校准选型。调研用于支撑实现决策，不改变产品主路径。

| 来源 | 用途 | 关键结论 |
| --- | --- | --- |
| OWASP Password Storage Cheat Sheet | KDF 选型与参数下限 | 使用 Argon2id；最低 m=19 MiB、t=2、p=1；参数随哈希一起保存以便后续升级。 |
| OWASP Cryptographic Storage Cheat Sheet | 加密模式与密钥管理 | 优先使用带认证的加密模式；用 KEK 包装 DEK（信封加密）；KDF 从口令派生 KEK 加密随机 DEK，改口令无需重加密数据；密钥须相互独立；使用 CSPRNG。 |
| libsodium pwhash（Argon2id） | KDF 参数档位与内存清除 | INTERACTIVE 64 MiB、MODERATE 256 MiB、SENSITIVE 1024 MiB；salt 固定长度随机；必须保存 opslimit/memlimit/alg；用 mlock/munlock 清除口令内存。 |
| libsodium XChaCha20-Poly1305 | AEAD 选型 | 192-bit nonce 允许安全使用随机 nonce；同一密钥可加密近乎无限条消息；combined 模式把认证标签附加在密文后；支持附加数据认证非机密头部。 |
| Android Keystore 官方文档 | 平台密钥绑定 | 密钥材料不可导出、可绑定 TEE/StrongBox；可要求用户认证后才可用；BiometricPrompt 配合 CryptoObject；可设置新生物识别注册时使密钥失效。 |

## 2. 密码学选型

最终库选择待实现评审确认。初始选型如下。

### 2.1 密钥派生（KDF）

- 算法：Argon2id。
- 用途：从主密码派生密钥加密密钥 KEK。
- salt：每个保险库独立随机生成，至少 16 字节（128-bit）。
- 输出：32 字节（256-bit）。
- 起始参数档案（待基准测试）：内存 m 以 64 MiB 为起点，迭代 t=3，并行 p=1；在目标低端设备上调到解锁约 1 秒内，且不得低于 OWASP 下限 m=19 MiB、t=2、p=1。
- 低内存设备允许降级到 OWASP 下限档，并把实际参数写入保险库头部。
- 必须把算法标识、版本、m、t、p、salt 写入头部，以便参数升级和重哈希判断。

### 2.2 认证加密（AEAD）

- 主选：XChaCha20-Poly1305。
- 理由：192-bit nonce 支持安全使用随机 nonce，降低 nonce 复用风险；适合本地文件和迁移包这类无集中计数器的场景；认证标签同时保护密文和头部附加数据。
- nonce：每次加密随机生成 24 字节（192-bit），绝不复用同一密钥下的 nonce。
- 附加数据（AAD）：把保险库头部（架构版本、cryptoProfile 等）作为附加数据认证，防止头部被篡改。
- 硬件回退：当平台或安全硬件要求时，允许使用 AES-256-GCM。GCM 使用 96-bit nonce 且对 nonce 复用敏感，因此仅作回退，并在头部记录所用 AEAD 标识。

### 2.3 随机源

- 所有密钥、salt、nonce 必须由 CSPRNG 生成。
- 不得使用非加密安全的伪随机数。
- Rust 核心使用加密安全随机源，平台层使用各自的安全随机 API。

### 2.4 加密库选型

**已定（2026-06-24，D-006/D-004）：采用 Rust 共享核心，加密原语用 Rust crate（RustCrypto 的 `argon2` + `chacha20poly1305`，或 libsodium 经 `libsodium-sys`）。** 不自行实现密码学原语；最终 crate 与版本在实现评审锁定，并以已知答案测试（第 12 节）验证。

下面保留当初对 Android 原生库的比较作背景（均**未采用**）：

- lazysodium-android（libsodium 的 Android 绑定）：同时提供 Argon2id 与 XChaCha20-Poly1305，恰好覆盖第 2.1、2.2 节；MPL-2.0；min SDK 24；但仅 Android、项目体量小。
- Google Tink（Apache-2.0）：提供 AEAD 等高层 API，但不提供基于口令的 KDF（Argon2id），需另配。
- Bouncy Castle（MIT 式）：Argon2id 与 ChaCha20-Poly1305 都有，但 API 偏底层。

## 3. 密钥层级与域分离

采用信封加密。主密码不直接加密保险库内容。

```text
主密码
  -> Argon2id(salt, 参数) -> KEK（密钥加密密钥，仅驻留内存）
随机生成 DEK（数据加密密钥，256-bit）
  -> 用 KEK 以 AEAD 包装为 wrappedDEK，随保险库保存
DEK
  -> 通过 KDF 域分离派生子密钥
       -> 保险库内容密钥
       -> 迁移包密钥
       -> 同步载荷密钥
```

规则：

- KEK 只在内存中存在，绝不落盘。
- DEK 随机生成，只以被 KEK 包装的形式保存。
- 改主密码只需用新 KEK 重新包装 DEK，不需要重加密整库。
- 保险库、迁移包、同步载荷使用相互独立的域分离子密钥，避免跨用途密钥重用。
- 各子密钥派生使用不同的上下文标签，确保用途隔离。
- 任意密钥使用后在可行时立即从内存清零。

## 4. 保险库文件格式

保险库文件由明文头部和加密载荷组成。头部本身不含秘密，但通过 AAD 受认证保护。

```text
VaultFile
- magic                 固定标识，用于识别文件类型
- formatVersion         文件格式版本
- schemaVersion         数据架构版本
- cryptoProfile
    - kdf: { id, version, m, t, p, salt }
    - aead: { id }
- wrappedDEK            被 KEK 包装的数据加密密钥及其认证标签
- nonce                 内容加密使用的随机 nonce
- ciphertext           加密后的保险库载荷
- authTag              内容认证标签
```

规则：

- magic、formatVersion、schemaVersion、cryptoProfile 作为 AAD 参与内容认证，任何篡改都会导致解密失败。
- 加密载荷覆盖所有有意义的内容：标题、标签、自定义字段、密码、TOTP 种子、备注。
- 解密前必须先校验认证标签，校验失败立即终止，不输出任何部分明文。
- 解析时应尽量保留未知字段，避免跨版本迁移丢失数据。
- 架构和格式版本不兼容时，先按迁移规则处理，再读取内容。

## 5. 解锁与会话模型

### 5.1 主密码解锁

```text
用户输入主密码
  -> Argon2id 用头部参数派生 KEK
  -> 用 KEK 解包 wrappedDEK 得到 DEK
  -> 解包失败即口令错误，记错误次数，不泄露细节
  -> 派生内容子密钥，解密保险库载荷
  -> 建立内存会话，持有解锁所需的最小密钥材料
```

### 5.2 生物识别解锁

- 生物识别是便利机制，不替代主密码。
- 平台安全存储包装本地解锁密钥：Android 使用 Keystore（TEE/StrongBox），iOS 使用 Keychain 与 Secure Enclave（部分细节待验证），HarmonyOS 能力待验证。
- 包装密钥要求用户认证后才可用，对应 Android 的用户认证授权（按时长或按操作）。
- 新增生物识别注册时使密钥失效，对应 Android 的相应设置，防止他人新录入的生物特征解锁。
- 必须允许用户关闭生物识别解锁，回退到主密码。

### 5.3 键盘会话授权

- 键盘只获得读取被选中字段所需的最小、短时会话授权，绝不持有主密码。
- 会话采用短生命周期令牌，超时、切换应用或锁屏后失效。
- 键盘锁定态不持有可解密内容的密钥材料。

### 5.4 会话与内存清除

- 解锁状态有空闲超时和显式锁定。
- 锁定、超时、进入后台时清除内存中的密钥和敏感值。
- 口令缓冲区使用内存锁定并在使用后清零，对应 libsodium 的 mlock/munlock。
- 敏感值插入后在可行时尽快清零。

## 6. 平台密钥库绑定

平台密钥库用于硬件绑定和便利增强，主密码仍是根秘密。

| 平台 | 安全存储 | 绑定方式 | 状态 |
| --- | --- | --- | --- |
| Android | Android Keystore（TEE/StrongBox） | 不可导出密钥包装本地解锁密钥；按时长或按操作要求用户认证；BiometricPrompt 配合 CryptoObject；新生物识别注册使密钥失效。 | 官方明确 |
| iOS | Keychain + Secure Enclave | LocalAuthentication 门控；Secure Enclave 保护密钥。 | 方向明确，细节待验证 |
| HarmonyOS | 平台安全存储 | 原生安全存储与生物识别绑定。 | 待验证 |

说明：

- StrongBox 支持 AES-256 和 HMAC-SHA256 等算法，但更慢、资源更受限，是否启用取决于威胁模型，作为可选增强。
- 密钥库操作不应在主线程执行，避免界面卡顿。
- 平台密钥库不改变信任边界：云端和网络仍不可接触明文。

## 7. 内存与日志卫生

- 不记录主密码、KEK、DEK、子密钥、密码字段、TOTP 种子和生成的验证码。
- 崩溃报告和诊断必须清理字段值和保险库标识符。
- 默认不启用遥测；若后续增加，必须用户选择加入并排除秘密。
- 敏感界面在平台支持时设置防截屏，对应 Android 的 FLAG_SECURE。
- 使用复制兜底时，剪贴板内容应在限定时间后清除，并在平台支持时标记为敏感。
- 密钥和敏感值使用后在可行时清零。

## 8. 迁移包与同步的密码学规则

- 迁移包和同步载荷使用各自的域分离子密钥，独立于本地保险库内容密钥。
- 迁移包和同步对象都必须带认证标签，先校验后使用。
- 导入失败不得覆盖上一份有效保险库，必须可回滚。
- 导入和同步在应用变更前校验格式与架构兼容性。
- 同步只处理密文；冲突元数据可对服务端可见，但不得暴露内容。

## 9. 密钥轮换与算法演进

- cryptoProfile 写入头部，使参数可随设备能力升级。
- 解锁时若检测到参数低于当前目标，可在用户下次解锁时用新参数重新派生并重新包装 DEK。
- 怀疑密钥泄露时支持轮换 DEK 并重加密内容。
- 预留 AEAD 和 KDF 算法的可替换路径，便于在出现新攻击时迁移。
- 保留旧参数读取能力，直到所有数据完成迁移。

## 10. 恢复边界

- 主密码是根秘密。主密码丢失意味着本地保险库无法恢复，这是设计取舍。
- 任何可选恢复机制必须单独设计并通过评审，不得削弱零知识属性。
- 云账号密码重置绝不解密保险库数据。
- 产品必须在界面上清楚说明这一恢复边界。

## 11. 威胁与缓解映射

| 威胁 | 缓解 |
| --- | --- |
| 设备存储被检查 | 全部内容用 AEAD 加密；KEK 仅驻留内存；密钥可绑定安全硬件。 |
| 弱主密码被暴力破解 | Argon2id 内存困难派生；参数随设备调优；提示强口令。 |
| 文件被篡改 | 头部作为 AAD 认证；解密前校验标签，失败即终止。 |
| nonce 复用 | 主选 XChaCha20-Poly1305 的大 nonce 支持随机 nonce；GCM 仅作回退并谨慎处理。 |
| 进程被攻破窃取密钥 | 平台密钥库密钥不可导出；要求用户认证；最小会话授权；使用后清零。 |
| 秘密泄漏到日志或崩溃报告 | 严格不记录策略；崩溃报告清理；默认不遥测。 |
| 导入损坏数据 | 认证校验、版本校验、失败回滚不覆盖有效库。 |
| 服务端或网络接触明文 | 仅上传密文；服务端不接收主密码和明文；恢复不解密数据。 |

## 12. 验证清单

- 已知答案测试：用固定测试向量验证 KDF 和 AEAD 实现一致。
- 错误口令：用错误主密码解包 DEK 必然失败，且不泄露区分信息。
- 篡改检测:修改头部或密文任意字节都导致解密失败。
- 参数持久化：头部保存的 KDF 参数被正确读取并用于解锁。
- 改口令：改主密码后旧库可用新口令解锁，且未重加密整库。
- 生物识别失效：新增生物识别注册后，受保护密钥按设计失效。
- 日志扫描：日志和崩溃报告不含任何秘密或字段值。
- 内存清除：解锁会话结束后敏感缓冲区被清零（在可观测范围内）。

## 13. 不在本文档范围与待验证

- 加密库已定走 Rust 共享核心（见第 2.4 节与技术架构第 2 节）；最终 Rust 加密 crate（RustCrypto 或 libsodium-sys）与版本在实现评审锁定。
- 移动端 Argon2id 具体参数，待目标设备基准测试。
- iOS Keychain/Secure Enclave 与 HarmonyOS 安全存储的具体集成细节，待平台验证。
- 可选恢复机制的具体设计，待单独评审。
